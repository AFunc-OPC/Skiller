# OpenSpec 页面视觉对齐设计文档

## 1. 背景

当前 OpenSpec 页面已经具备核心功能能力：

- 从项目详情进入项目级 OpenSpec 看板
- 展示项目状态、change 列表、详情区与 artifact 预览
- 提供 OpenSpec 终端面板辅助执行 `openspec init` 等命令

但从视觉语言和页面组织上看，这一页仍更像“功能已经拼齐的工作页”，还没有完全收敛为当前 Skiller 系统中的正式模块。

本次设计的目标不是重新定义 OpenSpec 页的功能，而是在现有能力边界内，把它调整为与当前系统中度一致、同时保留适度 OpenSpec 工作台识别度的页面。

## 2. 设计目标

本次视觉对齐设计的目标如下：

- 让 OpenSpec 页面在整体气质上明确属于当前 Skiller 系统
- 提升视觉语言一致性，尤其是按钮、卡片、面板、tabs 和状态表现
- 优化页面内部的信息组织和层次感
- 保留 OpenSpec 作为“专业工作台”的适度识别度
- 在不扩大功能范围的前提下，让终端面板成为更自然的系统内辅助工作区

## 3. 用户偏好约束

本次设计基于以下已确认偏好：

- 一致性强度：中度一致
- 当前问题优先级：视觉语言 > 信息组织 > 结构层次
- 参考页采用混合策略
- OpenSpec 专属识别度：中等
- 终端面板定位：半强调面板
- 页面节奏：编辑台式
- 改动范围：以 OpenSpec 页面为主，允许少量通用样式整理

## 4. 视觉方向

### 4.1 页面定位

OpenSpec 页面应被定义为“系统内的专业工作台”，而不是一套独立产品。

它应当满足以下三个条件：

- 比项目管理更具工具台感
- 比仓库管理更聚焦 change、artifact 和预览流
- 整体仍属于当前 `stage + glass-panel + muted border + restrained accent` 的系统语言

设计完成后，用户切入 OpenSpec 页面时，应感受到：

- 这是 Skiller 里的一个正式模块
- 这是一个专业工作台，而不是普通详情页
- 它有 OpenSpec 自己的工作流识别度，但没有风格跳出

### 4.2 视觉继承项

OpenSpec 页面应明确继承当前系统的以下特征：

- 外层 `workspace / stage / glass-panel` 语义
- 柔和边框和低对比背景层次
- 当前系统按钮的尺度、密度和反馈节奏
- 当前系统偏克制、偏专业的配色基调
- 面板之间以边框、背景差和留白建立层次，而非大色块切割

### 4.3 OpenSpec 专属识别项

OpenSpec 的识别度不应来自“完全不同的风格”，而应来自以下局部强化：

- 页头中的状态语义和项目上下文
- change 进度与 validation 信息的工作流表达
- artifact tabs 作为内容导航带
- 终端面板作为流程中的辅助工作区

## 5. 信息架构与分区重构

### 5.1 页头

页头应从单行功能区调整为更正式的模块头部，建议采用两层结构：

1. 第一层：返回、项目名、刷新、终端入口
2. 第二层：项目路径、OpenSpec 状态、change 统计

这样可以让页面从进入第一眼就建立主次关系，并减少当前“标题与内容区之间的视觉断裂”。

### 5.2 左栏：change 列表区

左栏应更像“状态面板”，而不是按钮堆叠区。

建议分为三段：

1. 顶部：filter tabs
2. 中部：change cards
3. 底部：当前筛选下的轻摘要或空状态

每个 change card 都应是一个有明确层级的信息单元，而不是简单的点击按钮。

建议层次：

- 第一行：change 名称
- 第二行：任务/状态摘要
- 第三行：validation 或风险提示

### 5.3 右栏：详情工作区

右栏应收束成正式内容面板，内部结构明确分为三层：

1. 顶部：当前 change 标题与局部刷新
2. 中部：artifact tabs
3. 主体：内容预览区

其中：

- 顶部承担上下文定位
- tabs 承担内容导航
- 主体承担阅读和预览

这样 Proposal / Design / Tasks / Specs 才会呈现出一致的“工作内容浏览区”语义。

### 5.4 终端面板

终端面板应调整为“有明确存在感的辅助工作区”，而不是临时附加块。

它仍然是辅助能力，不与 change 列表和详情区争抢主视觉重心，但应满足：

- 容器结构完整
- 与主页面风格一致
- 具有清晰的执行区语义

### 5.5 响应式行为

移动端无需重做交互模型，但需要维持清晰层次：

- 左右栏改为上下堆叠
- change 列表保持短区可滚动
- 详情区位于其下
- terminal panel 保持独立区块，不覆盖主工作流区域

## 6. 视觉语言细化

### 6.1 按钮体系

OpenSpec 页面中的按钮应收敛为三类：

1. 页面级动作按钮
2. 导航型按钮
3. 工具型小按钮

#### 页面级动作按钮

适用于：

- 返回项目
- 刷新
- 打开 OpenSpec 终端

这类按钮应延续当前系统主操作/次操作按钮语义。

#### 导航型按钮

适用于：

- change filter tabs
- artifact tabs

这类控件不应继续表现为普通按钮，而应更接近系统中的轻量切换器。

#### 工具型小按钮

适用于：

- 打开原文件
- 刷新当前 change

它们应在权重上弱于页面级动作。

### 6.2 卡片与面板层次

建议保留当前系统的柔和面板语言，但重新建立更清晰的层级：

- 外层工作区：系统级容器
- 左栏 / 右栏 / 终端：二级面板
- change card：三级信息单元

层次主要依赖：

- 背景层差
- 边框密度
- 结构留白

不依赖更重的阴影或更强的装饰。

### 6.3 Tabs 语义区分

filter tabs 与 artifact tabs 不应继续采用几乎相同的视觉表达。

建议区分为：

- 左侧 filter tabs：更轻、更工具化
- 右侧 artifact tabs：更稳、更像内容导航带

这样可以减少页面中“所有控件都长得类似”的问题。

### 6.4 状态色

状态色应保持克制，不引入新的高饱和视觉体系。

建议通过以下手段表达状态：

- 轻底色
- 边框强化
- 标签文本
- 进度或 validation 附近的局部强调

不使用大面积强色块作为主结构。

### 6.5 终端面板视觉处理

终端面板可以保留深色执行区语义，但必须服从系统结构语言：

- 继续使用系统内的圆角、边框和间距节奏
- 保持暗色，但避免看起来像另一套终端应用嵌入页面
- 输入区和输出区也应有明确的系统化容器关系

### 6.6 字体与节奏

本次不应为了 OpenSpec 页面单独建立新的字体气质。

更合理的做法是：

- 延续当前系统字体体系
- 通过字号层级、字重、留白和信息密度来建立“工作台感”

## 7. 改动范围

### 7.1 主要改动对象

本次改动应聚焦在 OpenSpec 页面及其直接相关区域：

- `OpenSpecBoardPage`
- `OpenSpecTerminalPanel`
- `osb-*` 样式体系
- 页头、左栏、右栏、tabs、terminal 的结构与视觉层级

### 7.2 允许的小范围共享整理

如果实现过程中发现有少量安全可复用的样式片段，可做极小范围共享，例如：

- 某个 tab 切换样式片段
- 某个轻量面板头部模式

但这类整理必须服务于本页，不应演变为全局样式重构。

### 7.3 明确不做的事

本次不包含：

- 重写全局 `glass-panel` 风格
- 反向改造项目管理、仓库管理、技能中心页面
- 重构全局导航壳层
- 仅为了视觉抽象而拆出大量新组件
- 调整 OpenSpec 页面的数据逻辑或功能边界

## 8. 验证标准

本次设计完成后，应满足以下四项标准：

1. 视觉连续性
从侧边栏切入 OpenSpec 页时，不会产生明显风格断层

2. 工作台感
相较项目管理页，OpenSpec 页明显更像一个 change / artifact / preview 协同区域

3. 信息主次
用户一眼能区分：
- 页头状态区
- 左侧 change 列表区
- 右侧详情内容区
- 终端辅助区

4. 风格克制
页面没有为了提升识别度而引入与当前系统冲突的新视觉系统

## 9. 预期结果

改造完成后，用户对 OpenSpec 页应形成以下感受：

- 更整
- 更像系统内正式模块
- 更像工作台
- 更不杂
- 终端不再像外挂块
- change list、tabs 和 preview 不再像基础按钮与容器的直接拼装

## 10. 结论

本次视觉对齐应被定义为：

在不改变 OpenSpec 页面核心功能边界的前提下，将其从“可用但视觉上偏临时组合的功能页”收敛为“与当前 Skiller 系统中度一致、具有专业工作台气质的正式模块页”。

该方向优先解决视觉语言一致性问题，同时改善信息组织与结构层次，并通过页头状态表达、artifact tabs、change 卡片和终端面板保留适度的 OpenSpec 专属识别度。
