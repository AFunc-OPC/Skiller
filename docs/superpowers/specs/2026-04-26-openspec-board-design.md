# OpenSpec 看板设计文档

## 1. 背景

当前应用已经具备项目管理与项目详情能力，项目详情以抽屉形式呈现，重点覆盖项目基础信息与项目技能管理。

本次需求是在现有“项目管理 -> 项目详情”路径中增加一个 OpenSpec 快捷入口，让用户可以从当前项目上下文直接进入该项目的 OpenSpec 可视化看板，查看 change 列表、进度、风险、关键产物文件，并在未初始化时通过交互式终端辅助完成 `openspec init`。

该设计以现有项目结构为前提：

- 项目详情入口位于 `src/pages/ProjectsPage.tsx`
- 前端通过 `src/api/project.ts` 调用 Tauri `invoke`
- 后端项目命令位于 `src-tauri/src/commands/project.rs`
- Rust 侧已经具备 shell/command 基础能力，可作为 OpenSpec CLI 封装基础

## 2. 目标

本次设计的目标如下：

- 在项目详情中提供进入当前项目 OpenSpec 看板的快捷入口
- 打开后进入独立全屏页面，而不是继续堆叠在详情抽屉内
- 以当前项目为上下文，展示 OpenSpec changes、进度、风险与关键产物
- 支持应用内只读预览 `proposal.md`、`design.md`、`tasks.md` 与 `specs/` 内容
- 支持一键在系统中打开对应原文件
- 第一版看板仅使用只读命令作为受控数据源
- 当项目未初始化 OpenSpec 时，提供可交互 bash 终端，默认引导 `openspec init`
- 为第二阶段扩展到工作台式看板保留结构

## 3. 非目标

第一版明确不包含以下内容：

- 在看板内直接编辑 OpenSpec 文件
- 在看板内直接执行 `verify`、`sync`、`archive` 等流程推进命令
- 将任意终端输出直接映射为看板主状态
- 构建跨项目的全局 OpenSpec 总览中心
- 做高频自动轮询或复杂后台持续同步

## 4. 用户价值与优先级

本需求的成功标准按优先级排序如下：

1. 能串起来：从项目详情入口，到看板总览，到文件预览，到初始化终端辅助，整条路径顺畅
2. 看得全：用户尽量不离开应用就能浏览大部分 OpenSpec 关键信息与产物
3. 看得清：用户进入页面后能快速理解当前项目有哪些 change、分别处于什么状态、卡在哪里

虽然有主优先级，但三者都需要在第一版内同时成立。

## 5. 方案选择

经过方案比较，最终采用“受控看板 + 受控命令聚合层 + 独立终端面板”的方案。

采用原因：

- 能保证看板的数据模型稳定，不直接依赖零散 bash 输出
- 第一版边界清楚，只开放只读命令，风险可控
- 终端作为辅助能力存在，不会把看板退化成嵌入式 shell
- 后续从第一版平滑扩展到工作台式看板时，无需推翻页面结构

未采用的方向：

- 不采用“前端直接消费原始命令结果”的快交付方案，因为它会削弱可视化一致性
- 不采用“一开始就搭完整工作台骨架并开放大量未来占位”的重方案，因为第一版应优先做好主链路

## 6. 信息架构

### 6.1 入口

在现有项目详情抽屉中新增快捷入口按钮，例如“打开 OpenSpec 看板”。

点击后进入一个独立的全屏 OpenSpec 页面，并自动带入当前 `projectId` 与项目路径上下文。

### 6.2 页面定位

该页面是“当前项目专属的 OpenSpec 看板”，不是全局 OpenSpec 中心。

页面中的以下能力都默认绑定到当前项目：

- change 查询
- 状态聚合
- 文件预览
- 原文件打开
- OpenSpec 终端默认工作目录

### 6.3 页面结构

全屏页分为三个主区域：

1. 顶部工具栏
2. 左侧 change 列表区
3. 右侧详情工作区

#### 顶部工具栏

用于承载页面级导航和状态信息：

- 返回项目页
- 显示当前项目名和项目路径
- 显示 OpenSpec 就绪状态
- 筛选 change 范围
- 手动刷新
- 打开 OpenSpec 终端

#### 左侧 change 列表区

用于展示当前项目的 changes：

- 默认显示未归档 change
- 支持切换查看 archived
- 展示 change 摘要、进度、风险与最近更新时间

#### 右侧详情工作区

用于展示当前选中 change 的详细内容：

- change 概览
- artifact 导航
- 文件预览
- 当前 change 级别的轻量操作

### 6.4 特殊状态页

特殊状态不额外跳转，直接在同一全屏页中呈现：

- CLI 不可用
- 项目未初始化 OpenSpec
- OpenSpec 已初始化但当前没有可展示的 change

## 7. 受控数据链路

### 7.1 设计原则

第一版采用两条链路分离：

1. 看板数据链路：只走受控命令与文件聚合，驱动页面状态
2. 交互终端链路：供用户手动执行 `openspec init` 或其他命令，但不直接充当页面主数据源

这可以避免终端行为与看板状态互相污染。

### 7.2 第一版允许的受控命令

第一版看板主数据只允许来自以下只读命令：

- `openspec list --changes --json`
- `openspec show <item> --type change --json`
- `openspec status --change <id> --json`
- `openspec validate <item> --type change --json`

当页面需要展示主规范或 `Specs` 视图中的规范详情时，同样允许以下只读命令进入受控链路：

- `openspec list --specs --json`
- `openspec show <item> --type spec --json`

环境与初始化状态判断需要辅助检查：

- `openspec --version`
- 当前项目路径下是否存在 `openspec/` 目录及关键结构

### 7.3 后端聚合层

Rust 侧不直接把单条 OpenSpec 命令裸暴露给前端，而是新增专门的 OpenSpec 聚合服务层。该层负责：

1. 在当前项目路径下执行受控命令
2. 解析 `--json` 输出并转换为稳定的页面 ViewModel
3. 补充文件系统信息
4. 汇总为前端可直接渲染的统一状态对象

建议补充的文件系统信息包括：

- `proposal.md`
- `design.md`
- `tasks.md`
- `specs/` 目录下的 spec 文件列表
- 关键文件的最近修改时间
- `tasks.md` 勾选完成度

### 7.4 顶层状态模型

页面顶层统一成四种状态：

- `cli_unavailable`：系统中没有可用 `openspec`
- `not_initialized`：项目目录可用，但没有 OpenSpec 关键结构
- `ready_empty`：OpenSpec 已就绪，但当前无可展示 change
- `ready`：正常展示看板

### 7.5 Change 级状态模型

每个 change 的可视化状态来自组合聚合，而不是单一命令：

- 基础信息来自 `list/show`
- artifact 流程进度来自 `status`
- 风险与校验信息来自 `validate`
- 文件存在性来自 change 目录内容
- 任务完成度来自 `tasks.md` 勾选解析
- 最近更新时间来自 change 目录或关键文件时间戳

### 7.6 一致性原则

为避免语义冲突，第一版统一采用以下规则：

- `status` 是流程进度的主语义来源
- `tasks.md` 勾选结果用于补充任务完成度
- 文件存在性只用于补充“产物是否存在/可预览”
- `validate` 结果只用于风险提示，不直接改写主进度状态

## 8. 交互设计

### 8.1 主流程

推荐主交互流：

1. 用户在项目详情中点击“打开 OpenSpec 看板”
2. 进入全屏页面后先进行环境判断
3. 页面根据状态展示未安装、未初始化、空状态或正常看板
4. 正常状态下默认选中第一个未归档 change
5. 用户在左侧切换 change，在右侧查看详情与文件预览
6. 用户可随时手动刷新
7. 用户可打开 OpenSpec 终端辅助执行初始化或其它命令

### 8.2 顶部工具栏

第一版建议包含：

- 返回按钮
- 页面标题：`<项目名> / OpenSpec`
- 项目路径简显
- 当前状态徽标
- change 视图切换：进行中 / 已归档 / 全部
- 刷新按钮
- 打开 OpenSpec 终端按钮

### 8.3 左侧 change 列表

每个 change 卡片建议展示：

- change 名称
- 状态标签
- artifact 进度摘要
- task 完成度，例如 `5/8 tasks`
- 最近更新时间
- 风险摘要，例如 warning、缺失文件、任务未完成

默认展示策略：

- 默认只看未归档 changes
- 支持切换查看 archived
- 排序按最近更新时间倒序

### 8.4 右侧详情区

右侧详情区分为三层：

1. change 概览条
2. artifact 导航条
3. 主内容区

#### change 概览条

展示：

- change 名称
- change 路径
- archived 状态
- 最近更新时间
- validate 摘要
- 打开 change 文件夹
- 刷新当前 change

#### artifact 导航条

建议包含：

- `Overview`
- `Proposal`
- `Design`
- `Tasks`
- `Specs`

其中：

- `Overview` 展示综合摘要
- 其他标签对应文件预览
- 文件缺失时展示明确缺失状态，不显示空白内容区

#### 主内容区

按标签显示：

- `Overview`：展示 `status` 进度、任务完成率、文件存在性、风险摘要
- `Proposal`：只读预览 `proposal.md`
- `Design`：只读预览 `design.md`
- `Tasks`：只读预览 `tasks.md`
- `Specs`：展示 spec 文件树与所选文件预览

## 9. 文件预览设计

### 9.1 预览范围

第一版应用内预览至少覆盖：

- `proposal.md`
- `design.md`
- `tasks.md`
- `specs/` 下的规范文件

### 9.2 预览体验

所有文件以只读方式展示，目标是快速浏览而非编辑。

每个预览视图建议显示：

- 文件名
- 相对路径
- 最近更新时间
- 打开原文件按钮

### 9.3 Specs 展示方式

`Specs` 视图采用文件树 + 预览区结构：

- 左侧列出 spec 文件树
- 右侧展示当前选中文件内容

第一版不提供编辑、保存、内联修改等能力。

## 10. OpenSpec 终端面板

### 10.1 定位

终端面板是看板的辅助操作区，而不是主页面常驻主区域。

建议使用底部抽屉或侧边面板形式，从当前看板页内唤起。

### 10.2 默认行为

终端面板默认工作目录为当前项目路径，并提供：

- 当前项目路径提示
- 预填命令：`openspec init`
- 常见补充参数提示，例如 `--tools`、`--profile`
- 切换到自由输入的能力

### 10.3 权限边界

终端面板的产品定位是“默认围绕 OpenSpec，但允许自由输入任意命令”。

因此第一版定义为：

- 默认是 OpenSpec 导向终端
- 允许用户输入任意命令
- 任意命令执行结果只停留在终端上下文，不自动进入看板主状态模型

### 10.4 与看板数据的关系

终端执行不会自动改写页面状态。用户执行命令后，需要通过以下方式让看板重新采集：

- 点击刷新
- 页面重新获得焦点触发轻量刷新

## 11. 刷新与同步策略

第一版采用混合刷新策略：

- 页面进入时做首次全量加载
- 用户可手动刷新
- 页面重新获得焦点时执行一次轻量刷新

第一版不做高频自动轮询，以减少命令开销与状态抖动。

## 12. 错误处理

### 12.1 环境错误

例如：

- `openspec` CLI 不存在
- shell 环境异常
- 当前项目路径不可访问

处理方式：

- 页面展示明确状态卡
- 显示原因说明
- 保留帮助信息与终端入口

### 12.2 初始化状态错误

例如：

- 存在 `openspec/` 目录但结构不完整
- 关键目录或文件缺失

处理方式：

- 归类为未就绪或未完整初始化
- 不误判为正常看板状态

### 12.3 Change 级错误

例如：

- 某个 change 可以列出，但 `show/status/validate` 某一步失败
- 某个 artifact 文件缺失或无法读取

处理方式：

- 页面整体仍保持可用
- 在该 change 卡片或详情区展示局部错误
- 避免单个异常拖垮整个页面

### 12.4 终端错误

例如：

- `openspec init` 执行失败
- 用户命令非法
- 命令中断

处理方式：

- 错误仅在终端面板中展示
- 主看板状态不直接受影响

## 13. 测试策略

### 13.1 Rust 聚合层测试

重点覆盖：

- CLI 可用性判断
- OpenSpec 初始化状态判断
- `list/show/status/validate` 的 JSON 解析
- `tasks.md` 勾选进度解析
- 文件存在性聚合
- 顶层状态与 change 状态汇总

### 13.2 前端组件测试

重点覆盖：

- 顶层状态分支渲染
- change 列表切换
- artifact 标签切换
- 文件缺失与错误提示
- 文件预览与打开原文件动作

### 13.3 页面流程测试

重点覆盖：

- 从项目详情进入看板
- 页面首次加载与刷新
- 未初始化时打开终端
- 终端操作后刷新看板获取最新状态

## 14. 第二阶段工作台式看板扩展说明

虽然第一版以轻量操作看板为目标，但结构需要为第二阶段工作台式看板保留扩展位。

第二阶段建议新增：

- 顶部工具栏中的受控流程命令入口：`verify`、`sync`、`archive`
- 命令执行结果面板
- 更完整的 change 下一步建议
- 更明确的流程阶段提示与最近一次命令执行记录

扩展原则：

- 保持“受控看板命令”和“自由终端”分层
- 工作台新增能力应建立在第一版聚合层之上，而不是回退为原始命令查看器
- 第二阶段允许加入写操作，但需要额外定义禁用条件、执行反馈和刷新一致性规则

## 15. 结论

本设计将 OpenSpec 能力引入现有项目详情路径，但不把复杂工作流直接压进详情抽屉，而是通过独立全屏看板承接当前项目的 OpenSpec 上下文。

第一版以“受控只读看板 + 文件预览 + 初始化辅助终端”为核心，优先打通从项目详情入口到 OpenSpec 可视化浏览与初始化辅助的完整链路，并为后续扩展到工作台式看板保留稳定结构。
